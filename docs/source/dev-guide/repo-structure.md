# Repository Structure

The workspace is a git repository with the following key files and directories:

- `features/`: [Feature Source](#feature-source) — canonical source tree for all feature files.
- `lib/`: [Shared Library](#shared-library) — canonical source for helper functions used by feature scripts.
- `test/`: [Tests](#tests) — scenario tests for features and unit tests for the shared library.
- `scripts/`: [Development Workflow](#development-workflow) — scripts for build, sync, validation, and other development routines.
- `.devcontainer/`: [Development Environment](#development-environment) — configurations for the development environment.
- `.github/workflows/`: [CI/CD Pipelines](#ci-cd-pipelines) — GitHub Actions workflows for continuous integration and deployment.
- `docs/`: [Documentation](#documentation) — Documentation source files and website configuration.


:::{admonition} Read-only directories
:class: danger

The following files and directories should **never be edited**; they are either fully auto-generated and git-ignored, symbolic links, or external dependencies:

- `src/`: [Feature Output](#feature-output) — publication-ready feature scripts generated from `features/` and `lib/`.
- `docs/features/`: [Feature Documentation](#feature-documentation) — User-facing feature references generated from `features/` metadata and notes.
- `dist/`: [Distribution Artifacts](#distribution-artifacts) — Packaged installers and related files generated by build scripts.
- `test/unit/bats/`: [Bats Testing Framework](#bats-testing-framework) — Git submodule containing the Bats testing framework for unit tests.
- `.devcontainer/_src/`: [Dev Container Source Link](#dev-container-source-link) — Symlink to `src/` for local development inside the dev container.
:::


## Feature Source

Each subdirectory of `features/` holds the source of one feature. The directory name must match the feature's `id` in its `metadata.yaml` file. Each feature directory contains:

- `metadata.yaml`: Feature metadata, including description, options, and dependencies.
- `install.bash`: **Body-only** installer script without the autogenerated header (bash ≥4.0).
- `NOTES.md`: Additional user-facing documentation for the feature.
- `files/`: Artifacts used by the feature, if any (e.g. config files, templates, static assets).
- `dev-notes/`: Developer-facing documentation (research, tool install methods, API/options narrative, design).
  - `dev-notes/feature.md`: Feature reference for implementers (installation methods, upstream behavior, options, citations).
  - `dev-notes/implementation.md`: Implementation notes, design decisions, edge cases, and maintenance guidance.

The `features/` directory also contains some source files shared across features:

- `bootstrap.sh`: A thin POSIX-compliant shim that ensures bash is available, then exec's the feature's `install.bash`.
- `features/derived-options.yaml`: Shared options that are injected into each feature at sync time.
- `features/metadata.schema.json`: JSON Schema for `metadata.yaml` files.
- `features/install-os-pkg/manifest.schema.json`: JSON Schema for `install-os-pkg` manifest files.


## Shared library

`lib/` contains the canonical source for the shared helper library, where each file is a module of related functions for a specific domain. These are copied into every feature's `src/*/_lib/`, which are then sourced by the feature scripts.


## Tests

Every feature that has scenarios has a matching subdirectory under `test/`.
See [Testing](testing.md) for the full guide.

`test/run.sh` is the unified test dispatcher. Its `feature <name>` subcommand wraps both the devcontainer CLI scenario tests and the fail scenarios in a single call. Other subcommands (unit, macos, dist, dry-run) delegate to the respective runners.

`test/run-fail-scenarios.sh` is the underlying runner for "expected-to-fail"
scenarios — cases where the feature should exit non-zero (e.g. invalid
inputs). It is invoked by `test/run.sh feature` and by CI.

## Development Workflow

### The bootstrap pattern

The devcontainer specification guarantees only POSIX `sh` is available when
`install.sh` runs. Bash 4+ features used by the library are therefore not
available at that point. The bootstrap pattern resolves this:

- `features/bootstrap.sh` is a POSIX `sh` script. Its only job is to ensure `bash` is
  present and then `exec bash install.bash "$@"`.
- `install.bash` is written in bash ≥4 and uses all library features
  freely.

`features/bootstrap.sh` knows how to install bash on every supported distro (APT, APK,
DNF, microdnf, YUM, Zypper, Pacman). It is the single source of truth — one
copy is distributed to every feature root as `install.sh`.

### The sync mechanism

`scripts/sync-src.py` automates the distribution of shared files:

```
features/bootstrap.sh   →  src/*/install.sh           (one copy per feature)
lib/           →  src/*/_lib/                (one copy per feature)
features/*/install.bash  →  src/*/install.bash  (header prepended)
features/*/metadata.yaml →  src/*/devcontainer-feature.json
features/*/metadata.yaml →  src/*/dependencies/*.yaml
features/*/files/        →  src/*/files/       (copied, not symlinked)
```

It auto-discovers features by finding every `metadata.yaml` under
`features/`. No list is hard-coded.

```bash
# Sync all features
python3 scripts/sync-src.py
# Verify copies are up to date (exits non-zero if stale)
python3 scripts/sync-src.py --check
```

CI runs `python3 scripts/sync-src.py` as the first step of every workflow job to ensure
the working tree is consistent before building or testing.

### Lefthook (optional)

[Lefthook](https://github.com/evilmartians/lefthook) is configured in `lefthook.yml`. **The pre-commit hooks that run sync, shfmt, and shellcheck are currently commented out** in that file, so they do not run unless you restore them.

The devcontainer runs `lefthook install` on create so hooks are ready if you enable them. For shell style and CI behavior, see [`docs/snippets/code-style.md`](../snippets/code-style.md) and [CI](ci.md).

## Feature Output

The entire `src/` directory is a **generated artefact** — it is git-ignored and regenerated by `scripts/sync-src.py`.

`src/` is fully auto-generated and git-ignored.

This is the canonical source for the install script's entry point, and is copied into each feature's `src/*/install.sh` by `scripts/sync-src.py`.


- `src/`: **Fully auto-generated and gitignored** — assembled from `features/` by `scripts/sync-src.py`. Consumed by the devcontainer CLI and build tooling. **Never edit files here directly.**
  - `src/*/`: Per-feature generated directory.
    - `src/*/devcontainer-feature.json`: Generated from `features/*/metadata.yaml` by `scripts/sync-src.py`.
    - `src/*/install.sh`: Generated copy of `features/bootstrap.sh` (POSIX shim). Overwritten on every sync.
    - `src/*/install.bash`: Generated assembled installer — autogenerated header + body from `features/*/install.bash`. Overwritten on every sync.
    - `src/*/_lib/`: Generated copy of `lib/`. Overwritten on every sync.
    - `src/*/dependencies/`: Generated from the `_dependencies:` key in `features/*/metadata.yaml` by `scripts/sync-src.py`.
    - `src/*/files/`: Copied from `features/*/files/` by `scripts/sync-src.py` (only for features that have a `files/` directory).


```
src/<feature>/
├── devcontainer-feature.json   ← Generated from metadata.yaml (git-ignored)
├── install.sh                  ← Generated bootstrap/POSIX shim (git-ignored)
├── install.bash                ← Generated: header + body from features/ (git-ignored)
├── _lib/                       ← Generated copy of lib/ (git-ignored)
├── dependencies/               ← Generated from metadata.yaml _dependencies: key
│   └── base.yaml               ← OS packages required before install.bash runs
└── files/                      ← Copied from features/<feature>/files/ (git-ignored)
```

The devcontainer CLI calls `install.sh` (the bootstrap) at image build time.
The bootstrap ensures `bash` is available, then execs
`install.bash "$@"`, which is the real installer written in bash ≥4.
`install.bash` sources helpers from `_lib/` — the local copy
of the shared library.



Reasons for not committing:
- They are exact copies — committing them would create merge conflicts and
  make diffs noisy every time a library file is touched.
- CI regenerates them from the canonical sources on every run, so the working
  tree is always correct.
- The git-ignore makes it immediately apparent when someone accidentally edits
  a generated copy (the change disappears on the next sync).






## Code style — `shfmt` and `shellcheck`

All shell scripts in this repository are formatted with
[shfmt](https://github.com/mvdan/sh) and linted with
[shellcheck](https://www.shellcheck.net/).

### Style configuration

`.editorconfig` is the single source of truth for shfmt style. The key
settings for `*.sh` and `*.bats` files:

| Setting | Value | Effect |
|---|---|---|
| `indent_size` | `2` | Two-space indentation |
| `switch_case_indent` | `true` | Case branches indented inside `case` blocks |
| `function_next_line` | `false` | Opening brace on the same line as `fn() {` |
| `space_redirects` | `true` | Space between redirect operator and target |
| `binary_next_line` | `false` | `&&` / `\|\|` at end of line (not start) |

`*.bats` uses `shell_variant = bats` so shfmt applies bats-aware parsing.

### Shellcheck configuration

`.shellcheckrc` sets global defaults:

```ini
shell=bash          # default dialect for files without a shebang
external-sources=true   # follow source/. directives
```

Per-file or per-line overrides use inline directives:

```bash
# shellcheck disable=SC2034
```

### Developer workflow

The [justfile](../../justfile) provides convenience recipes:

```bash
just format          # auto-format all shell files in place (shfmt -w)
just format-check    # check formatting without writing (used in CI)
just lint            # run shellcheck on all tracked .sh/.bash files
just sync            # regenerate _lib/ copies and install.sh stubs
```

VS Code users get formatting automatically via the
`foxundermoon.shell-format` extension (format on save) and inline lint
diagnostics via `timonwong.shellcheck`. Recommended extensions are listed in
`.vscode/extensions.json` and will be suggested when you open the repo.

### CI enforcement

Lint (shfmt + shellcheck) runs as part of the reusable **`ci.yaml`** workflow when the change set triggers the `lint` job (see [CI](ci.md)). Locally, use `just format-check` and `just lint`.

`shfmt --apply-ignore` respects `.editorconfig` / ignore rules so generated paths under `src/` are not formatted as hand-written sources.

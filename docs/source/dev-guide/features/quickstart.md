# Quickstart Guide

Each subdirectory of `features/` holds the source of one feature. The directory name is the canonical feature `id` (inferred by tooling); `metadata.yaml` does not define an `id` field. Each feature directory contains:

- `metadata.yaml`: Feature metadata, including description, options, and dependencies.
- `install.bash`: **Body-only** installer script without the autogenerated header (bash ≥4.0).
- `NOTES.md`: Additional user-facing documentation for the feature.
- `files/`: Artifacts used by the feature, if any (e.g. config files, templates, static assets).
- `dev-notes/`: Developer-facing documentation (research, tool install methods, API/options narrative, design).
  - `dev-notes/feature.md`: Feature reference for implementers (installation methods, upstream behavior, options, citations).
  - `dev-notes/implementation.md`: Implementation notes, design decisions, edge cases, and maintenance guidance.

The `features/` directory also contains some source files shared across features:

- `install.sh`: A thin POSIX-compliant shim that ensures bash is available, then exec's the feature's `install.bash`.
- `features/shared-options.yaml`: Shared options that are injected into each feature at sync time.
- `features/metadata.schema.json`: JSON Schema for `metadata.yaml` files.
- `features/install-os-pkg/manifest.schema.json`: JSON Schema for `install-os-pkg` manifest files.

```
features/<feature-id>/
├── metadata.yaml               ← Options, description, dependencies, version (source of truth)
├── install.bash                ← Body-only installer (bash ≥ 4; header auto-generated by sync)
├── files/                      ← Optional static files deployed by the feature
├── NOTES.md                    ← User-facing documentation (installed into the website)
└── dev-notes/                  ← Developer-facing notes
    ├── feature.md              ← Installation methods, upstream behavior, API notes
    └── implementation.md       ← Design decisions, edge cases, maintenance guidance
```

## Quick-start checklist

1. **Before writing any logic** — read the [shared library reference](#shared-library-reference) below. Many common operations (platform detection, GitHub API calls, checksum verification, user resolution, shell changes) are already implemented. Use library functions instead of reinventing them.
2. Create `features/<feature-id>/metadata.yaml` following the [feature metadata](#feature-metadata) section below and validating against [`features/metadata.schema.json`](../../features/metadata.schema.json).
3. Create `features/<feature-id>/install.bash` (body only — no header; the header is generated by `scripts/sync-src.py`) following the [install script](#install-script) section.
4. Run `just sync-src` to assemble `src/<feature-id>/` (generated `install.bash`, `install.sh`, `_lib/`, `devcontainer-feature.json`).
5. Create `test/<feature-id>/scenarios.json` and at least one `<scenario>.sh`. See {doc}`testing`.
6. If the feature needs OS packages before `install.bash` runs, add a `_dependencies.base` key to `metadata.yaml`.

## Sync

After creating or editing `install.bash`, run:

```bash
python3 scripts/sync-src.py
```

This assembles `src/<feature-id>/install.bash` (header + body), generates
`install.sh` (bootstrap) and `_lib/` for your new feature.
The entire `src/` directory is git-ignored — never commit anything there.

[Lefthook](https://github.com/evilmartians/lefthook) can run sync on commit if you enable the corresponding commands in `lefthook.yml` (they are **commented out** in the repo default).

Verify the sync is up to date before pushing:

```bash
python3 scripts/sync-src.py --check
```

After running `just sync-src`, the assembled feature lives in `src/<feature-id>/` (git-ignored):

```
src/<feature-id>/
├── devcontainer-feature.json   ← Generated from metadata.yaml
├── install.sh                  ← Generated bootstrap (DO NOT EDIT)
├── install.bash                ← Generated: header + body (DO NOT EDIT)
├── _lib/                       ← Synced library copy (DO NOT EDIT)
├── dependencies/               ← Generated from metadata.yaml _dependencies: key
└── files/                      ← Copied from features/<feature-id>/files/
```

# Introduction

SysSet treats system setup the way modern application runtimes treat deployments: as a **declarative, reproducible, portable** operation. A feature should produce the same result whether it runs inside a dev container during image build, inside a fresh VM provisioned by a CI job, or on a developer's laptop — and it should let you pin a snapshot, inspect it, mirror it, and replay it later. Four concrete commitments come out of that philosophy:

1. **Findability.** Features are published to both [GitHub Container Registry](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry) and [GitHub Releases](https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases). All distribution channels are built from the same bits on every release, so they never drift.
2. **Accessibility.** Features can be installed in any POSIX-compliant environment, including physical machines (e.g. to set up a new laptop or update a PC), virtual machines (e.g. to set up a cloud VM such as a GitHub Actions runner), containers (e.g. as a RUN instruction in a Dockerfile or executed inside a running container), and Dev Containers — with the same behavior across all platforms and installation methods. Every installer works on Debian/Ubuntu, RHEL/Fedora, Alpine, Arch, openSUSE, and macOS. Package-manager detection, user resolution, and shell-integration are all handled internally, so you do not pick a variant per OS.
3. **Interoperability.** Every feature is fully compliant with the [Dev Container Features Specification](https://containers.dev/implementors/features/) and can be used as a drop-in feature in any `devcontainer.json` file, where it is automatically installed by compliant tools like VS Code Dev Containers and GitHub Codespaces. At the same time, every feature can also be installed outside of containers with the same behavior, either using SysSet or directly from their release artifacts. Each installer accepts options both as environment variables (the way dev container tooling injects them) and as CLI flags. This is what lets the same installer run under `devcontainer up`, under SysSet's installer, and as a pipe-to-`sh` one-liner — with identical behavior.
4. **Reproducibility.** Every release is an immutable artifact, with a unique version and a single source of truth. You can always go back to a previous version, inspect its contents, mirror it to another registry, and install it with the same behavior as when it was first released. Features are designed to be as idempotent as possible, so re-running an installer after fixing an environment issue will not cause unintended side effects and does not require uninstalling first.

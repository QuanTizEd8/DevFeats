# Repository Structure

## Directory layout

| Path | Role |
|------|------|
| `features/` | **Source of truth** for all features. Each subdirectory is one feature. Also contains shared source files (`bootstrap.sh`, `shared-options.yaml`, schemas). |
| `lib/` | **Shared bash library** — canonical source for helper modules, copied into each feature's `_lib/` at sync time. |
| `src/` | **Generated output** (git-ignored) — assembled by `scripts/sync-src.py`; consumed by the devcontainer CLI and build tooling. Never edit files here. |
| `test/` | Test suite — scenario tests per feature, bats unit tests for `lib/`, macOS native tests, dist tests. |
| `scripts/` | Developer scripts — `sync-src.py`, `build-artifacts.sh`, `gen_docs.py`, release helpers. |
| `docs/` | Sphinx documentation — source under `docs/source/`, built site under `docs/.build/`. |
| `.github/workflows/` | GitHub Actions — `cicd.yaml`, `ci.yaml`, `cd.yaml`, `devcontainer.yaml`. |
| `.devcontainer/` | Dev container configs — main dev container + per-feature containers for interactive testing. |
| `bundles/` | Bundle metadata for grouped publishing. |
| `dist/` | **Generated** release artifacts (git-ignored). |
| `.local/` | Git-ignored scratch space — logs, test artifacts, temporary files. |

Root-level files:

| File | Role |
|------|------|
| `install.sh` | POSIX sh bootstrap for the standalone installer |
| `install.bash` | Bash ≥ 4 standalone installer (feature and manifest mode) |
| `justfile` | Developer recipes — run `just --list` |
| `.editorconfig` | shfmt style config |
| `.shellcheckrc` | shellcheck defaults |
| `lefthook.yml` | Lefthook hooks (pre-commit hooks are currently commented out) |

## Feature source (`features/`)

Each subdirectory of `features/` holds the source of one feature. The directory name is the canonical feature `id`. Each feature directory contains:

- **`metadata.yaml`** — feature metadata: description, options, dependencies, version. Source of truth for the generated `devcontainer-feature.json`.
- **`install.bash`** — body-only installer script (bash ≥ 4; no autogenerated header).
- **`NOTES.md`** — user-facing documentation (installed into the website).
- **`files/`** — static files deployed by the feature (config files, templates, scripts).
- **`dev-notes/`** — developer-facing notes: research, implementation decisions, edge cases.
  - `dev-notes/feature.md` — installation methods, upstream behavior, API notes.
  - `dev-notes/implementation.md` — design decisions, edge cases, maintenance guidance.

Shared files at the `features/` root:

- **`bootstrap.sh`** — POSIX sh shim: ensures bash is present, then `exec bash install.bash "$@"`. Copied to each `src/*/install.sh` at sync time. Never write it manually.
- **`shared-options.yaml`** — options injected into every feature at sync (`log_level`, `log_file`, `keep_cache`, …). Do not re-declare these in per-feature `metadata.yaml` files.
- **`metadata.schema.json`** — JSON Schema for `metadata.yaml`.
- **`install-os-pkg/manifest.schema.json`** — JSON Schema for ospkg manifest files.

## Generated output (`src/`)

The entire `src/` directory is a generated artefact — git-ignored, always regenerated by `scripts/sync-src.py`.

```
src/<feature>/
├── devcontainer-feature.json   ← Generated from metadata.yaml
├── install.sh                  ← Generated copy of features/bootstrap.sh
├── install.bash                ← Generated: autogenerated header + body from features/*/install.bash
├── _lib/                       ← Generated copy of lib/
├── dependencies/               ← Generated from metadata.yaml _dependencies: key
│   └── base.yaml               ← OS packages required before install.bash runs
└── files/                      ← Copied from features/<feature>/files/
```

**Never edit files in `src/`** — they are overwritten on every sync. Reasons `src/` is not committed:
- It consists of exact copies; committing creates noisy diffs and merge conflicts.
- CI regenerates from canonical sources on every run, so the working tree is always correct.
- The git-ignore makes it obvious when someone accidentally edits a generated file.

## Shared library (`lib/`)

`lib/` is the canonical source for the shared bash helper library. Each file is a module of related functions for a specific domain:

| Module | Domain |
|--------|--------|
| `os.sh` | Kernel, architecture, platform, and distro detection |
| `logging.sh` | Structured logging with emoji markers, log capture, cleanup |
| `net.sh` | Retrying downloads with curl/wget |
| `ospkg.sh` | Cross-distro package manager abstraction; ospkg manifest runner |
| `shell.sh` | Shell startup file paths, idempotent block writes, PATH exports |
| `json.sh` | JSON parsing without dedicated tooling |
| `github.sh` | GitHub API: releases, tags, asset downloads |
| `checksum.sh` | SHA-256 verification (sha256sum / shasum) |
| `users.sh` | User resolution, login shell changes |
| `str.sh` | String and argv helpers |
| `git.sh` | Shallow git clone |

Each module uses a guard variable (`_LIB_<NAME>_LOADED`) to prevent double-sourcing. See {doc}`writing-features` for the full API reference.

## Tests (`test/`)

```
test/
├── unit/                         ← bats unit tests for lib/ modules
│   ├── *.bats                    one file per lib/ module
│   ├── helpers/
│   │   ├── common.bash           reload_lib() + bats library loading
│   │   └── stubs.bash            create_fake_bin() / prepend_fake_bin_path()
│   ├── setup_suite.bash          bash ≥ 4 guard
│   └── bats/                     git submodules — never edit
├── <feature>/                    ← devcontainer CLI scenario tests
│   ├── scenarios.json            test matrix
│   ├── <scenario>.sh             assertion script (runs inside container)
│   ├── <scenario>/               optional build context (Dockerfile + files)
│   ├── linux/                    Linux-native scenarios (plain Docker)
│   │   ├── <scenario>.sh         assert.sh-based script
│   │   └── <scenario>.conf       optional: IMAGE, NETWORK, SETUP_CMD, RUN_AS
│   └── macos/                    macOS native scenarios (no Docker)
│       └── <scenario>.sh
├── lib/
│   └── assert.sh                 check() / fail_check() / reportResults()
└── install-os-pkg/
    └── dry-run/                  manifest resolution tests (plain Docker)
```

See {doc}`testing` for the full testing guide.

## Sync mechanism

`scripts/sync-src.py` assembles `src/` from the canonical sources:

```
features/bootstrap.sh    →  src/*/install.sh           (one copy per feature)
lib/                     →  src/*/_lib/                (one copy per feature)
features/*/install.bash  →  src/*/install.bash         (autogenerated header prepended)
features/*/metadata.yaml →  src/*/devcontainer-feature.json
features/*/metadata.yaml →  src/*/dependencies/*.yaml
features/*/files/        →  src/*/files/
```

Auto-discovers features by finding every `metadata.yaml` under `features/` — no list is hard-coded.

```sh
just sync        # regenerate src/
just sync-check  # verify copies are up to date (exits non-zero if stale)
```

CI runs `python3 scripts/sync-src.py` as the first step of every workflow job.

## Code style

All shell scripts are formatted with [shfmt](https://github.com/mvdan/sh) and linted with [shellcheck](https://www.shellcheck.net/).

### shfmt configuration (`.editorconfig`)

| Setting | Value | Effect |
|---------|-------|--------|
| `indent_size` | `2` | Two-space indentation |
| `switch_case_indent` | `true` | Case branches indented inside `case` blocks |
| `function_next_line` | `false` | Opening brace on the same line as `fn() {` |
| `space_redirects` | `true` | Space between redirect operator and target |
| `binary_next_line` | `false` | `&&`/`\|\|` at end of line |

`*.bats` files use `shell_variant = bats` for bats-aware parsing.

### shellcheck configuration (`.shellcheckrc`)

```ini
shell=bash            # default dialect for files without a shebang
external-sources=true # follow source/. directives
```

Per-file or per-line overrides use inline directives: `# shellcheck disable=SC2034`.

### Workflow

```sh
just format       # auto-format all shell files in place
just format-check # check formatting without writing (used in CI)
just lint         # run shellcheck on all tracked .sh/.bash/.bats files
```

`features/*/install.bash` files are **body-only** (no autogenerated header). Run `just sync` before `just lint` when `src/` is missing or stale — linting targets the assembled `src/*/install.bash` files.

VS Code users get formatting automatically via the `foxundermoon.shell-format` extension and inline lint diagnostics via `timonwong.shellcheck`. Recommended extensions are listed in `.vscode/extensions.json`.

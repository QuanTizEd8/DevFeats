# Writing Features

This guide covers how to create and maintain a devcontainer feature in this repository: required files, conventions, the shared bash library, and the full API reference.

---

## Quick-start checklist

1. **Before writing any logic** — read the [shared library reference](#shared-library-reference) below. Many common operations (platform detection, GitHub API calls, checksum verification, user resolution, shell changes) are already implemented. Use library functions instead of reinventing them.
2. Create `features/<feature-id>/metadata.yaml` following the [feature metadata](#feature-metadata) section below and validating against [`features/metadata.schema.json`](../../features/metadata.schema.json).
3. Create `features/<feature-id>/install.bash` (body only — no header; the header is generated by `scripts/sync-src.py`) following the [install script](#install-script) section.
4. Run `just sync` to assemble `src/<feature-id>/` (generated `install.bash`, `install.sh`, `_lib/`, `devcontainer-feature.json`).
5. Create `test/<feature-id>/scenarios.json` and at least one `<scenario>.sh`. See {doc}`testing`.
6. If the feature needs OS packages before `install.bash` runs, add a `_dependencies.base` key to `metadata.yaml`.

---

## Feature anatomy

```
features/<feature-id>/
├── metadata.yaml               ← Options, description, dependencies, version (source of truth)
├── install.bash                ← Body-only installer (bash ≥ 4; header auto-generated by sync)
├── files/                      ← Optional static files deployed by the feature
├── NOTES.md                    ← User-facing documentation (installed into the website)
└── dev-notes/                  ← Developer-facing notes
    ├── feature.md              ← Installation methods, upstream behavior, API notes
    └── implementation.md       ← Design decisions, edge cases, maintenance guidance
```

After running `just sync`, the assembled feature lives in `src/<feature-id>/` (git-ignored):

```
src/<feature-id>/
├── devcontainer-feature.json   ← Generated from metadata.yaml
├── install.sh                  ← Generated bootstrap (DO NOT EDIT)
├── install.bash                ← Generated: header + body (DO NOT EDIT)
├── _lib/                       ← Synced library copy (DO NOT EDIT)
├── dependencies/               ← Generated from metadata.yaml _dependencies: key
└── files/                      ← Copied from features/<feature-id>/files/
```

## Feature metadata

`features/<feature-id>/metadata.yaml` is the source of truth for:

1. The generated `devcontainer-feature.json` consumed by the devcontainer CLI.
2. The autogenerated `install.bash` header (argument parsing, help text, library sourcing).
3. OS package dependency manifests (`dependencies/*.yaml`).
4. Website documentation.

It follows the [JSON schema for `devcontainer-feature.json`](https://raw.githubusercontent.com/devcontainers/spec/refs/heads/main/schemas/devContainerFeature.schema.json) with the following additions and restrictions.

### Restrictions

Derivable fields such as `id`, `documentationURL`, and `licenseURL` are **prohibited** in `metadata.yaml` — they are generated at sync time. This ensures consistency and eliminates duplication.

### Additional fields

- **`_long_description`** — a detailed description used for documentation and help text.
- **`_dependencies`** — install-time and runtime dependency groups, defined as cross-platform ospkg manifests.

### Options

Each option becomes both a CLI flag (`--<option_name>`) and an environment variable (`<OPTION_NAME>`) injected by devcontainer tooling at build time. Option names use snake_case.

Supported types: `"string"`, `"boolean"`, and the SysSet-internal **`"array"`** type (serialized as `type: string` in the generated JSON). See {doc}`/guide/options` for how the array type works across all invocation channels.

For string options, two properties control how supporting tools present the allowed values:

| Property | Behaviour |
|----------|-----------|
| `"enum"` | **Strict** — user must choose one of the listed values. Use for a closed set. |
| `"proposals"` | **Suggestive** — listed values appear as suggestions; any value is accepted. |

```yaml
# Closed set — only these three values are accepted
mode:
  type: string
  default: fast
  enum: [fast, safe, dry_run]
  description: Operating mode.

# Open set — suggests common versions; any semver is valid
version:
  type: string
  default: latest
  proposals: [latest, "3.12", "3.11", "3.10"]
  description: Version to install.
```

### Shared options

`features/shared-options.yaml` defines options injected into every feature at sync time (`log_level`, `log_file`, `keep_cache`, `keep_build_deps`, …). **Do not re-declare these in per-feature `metadata.yaml` files.**

### Feature dependencies

Declare `dependsOn` when your feature requires another feature to have run first at build time:

```yaml
dependsOn:
  ghcr.io/|{{github_user}}|/|{{github_repo}}|/setup-user: {}
```

The devcontainer CLI and `install.bash` both resolve install order automatically from these edges.

### OS package dependencies

Declare `_dependencies.base` when the installer needs packages (e.g. `curl`, `ca-certificates`) present before any other work begins:

```yaml
_dependencies:
  base:
    packages:
      - ca-certificates
      - curl
```

`sync-src.py` converts this to `src/<feature>/dependencies/base.yaml` automatically. Call it at the top of `install.bash`:

```bash
ospkg__run --manifest "${_BASE_DIR}/dependencies/base.yaml" --skip_installed
ospkg__clean
```

### Validation and sync

```sh
just sync        # validate metadata, regenerate src/, check for stale files
just sync-check  # verify copies are up to date (exits non-zero if stale)
```

## Bootstrap pattern

The devcontainer spec guarantees only POSIX `sh` when `install.sh` runs — bash is not guaranteed. The bootstrap resolves this in two steps:

1. **`install.sh`** (POSIX sh) — checks whether `bash` is present; if not, installs it via the system package manager (`apk`, `apt-get`, `dnf`, `microdnf`, `yum`, `zypper`, `pacman`, Homebrew). Then `exec bash install.bash "$@"` to hand off.
2. **`install.bash`** (bash ≥ 4) — the real installer with full access to the shared library.

`install.sh` at each feature root is generated from `features/bootstrap.sh` by `scripts/sync-src.py`. It is identical across all features. **Never write it manually.**

## Install script

`features/<feature-id>/install.bash` contains the **body only** — no shebang, no `set -euo pipefail`, no path setup. The autogenerated header prepended by `sync-src.py` provides all of that. The sections below describe the conventions for the body.

### Library sourcing

Source only the libraries you need. `ospkg.sh` sources `os.sh` and `net.sh` automatically; all others must be sourced explicitly.

```bash
# ospkg.sh pulls in os.sh and net.sh — source it first.
# shellcheck source=_lib/ospkg.sh
. "$_SELF_DIR/_lib/ospkg.sh"
# shellcheck source=_lib/logging.sh
. "$_SELF_DIR/_lib/logging.sh"

# Source additional modules only when their functions are needed:
# shellcheck source=_lib/github.sh
. "$_SELF_DIR/_lib/github.sh"
# shellcheck source=_lib/checksum.sh
. "$_SELF_DIR/_lib/checksum.sh"
# shellcheck source=_lib/users.sh
. "$_SELF_DIR/_lib/users.sh"
# shellcheck source=_lib/shell.sh
. "$_SELF_DIR/_lib/shell.sh"
# shellcheck source=_lib/json.sh
. "$_SELF_DIR/_lib/json.sh"
# shellcheck source=_lib/str.sh
. "$_SELF_DIR/_lib/str.sh"
# shellcheck source=_lib/git.sh
. "$_SELF_DIR/_lib/git.sh"
```

> **Check the library first.** Before writing inline logic, confirm there is no existing function: use `os__kernel`/`os__arch` instead of `uname -s`/`uname -m`; `os__font_dir` instead of manual platform detection; `github__latest_tag` instead of a hand-rolled API call; `checksum__verify_sha256_sidecar` instead of inline sha256sum logic; `users__resolve_list` + `users__set_login_shell` instead of manual deduplication and `chsh` calls.

### Logging setup

Always set up logging immediately after sourcing the library, with the EXIT trap to ensure cleanup runs even on error:

```bash
logging__setup
logging__entry "My Feature"
trap 'logging__cleanup' EXIT
```

`logging__setup` redirects stdout and stderr through `tee` into a temp file. `logging__cleanup` (called on EXIT) flushes the captured output to `$LOG_FILE` if set. Call `logging__set_level` after parsing options so `--log_level` takes effect.

### Dual-mode argument parsing

Every feature must work in two modes:

- **Devcontainer mode** (no CLI arguments): options injected as environment variables.
- **CLI mode** (arguments present): called as `install.bash --version 1.2.3 --log_level trace`.

```bash
if [ "$#" -gt 0 ]; then
  # Reset all variables before parsing — avoid inheriting caller's env.
  VERSION=""
  LOG_LEVEL=""
  LOG_FILE=""

  while [[ $# -gt 0 ]]; do
    case $1 in
      --version)   shift; VERSION="$1"; shift ;;
      --log_level) shift; LOG_LEVEL="$1"; shift ;;
      --log_file)  shift; LOG_FILE="$1"; shift ;;
      --help|-h)   __usage__; exit 0 ;;
      --*)         logging__error "Unknown option: '${1}'"; exit 1 ;;
      *)           logging__error "Unexpected argument: '${1}'"; exit 1 ;;
    esac
  done
else
  logging__info "Reading environment variables."
  [ "${VERSION+defined}" ] && logging__read "Argument 'version': '${VERSION}'"
fi
```

The `[ "${VAR+defined}" ]` pattern (without `-z`) checks whether the variable was **set**, even if set to an empty string — distinguishing "caller set it empty" from "caller did not set it at all".

Include a `__usage__()` function that prints the full option list to stderr and is invoked for `--help`.

### Applying defaults

Apply defaults after argument parsing using `:-`:

```bash
: "${VERSION:=latest}"
: "${LOG_LEVEL:=info}"
: "${LOG_FILE:=}"
logging__set_level   # call after defaults so log_level takes effect
```

### Validation

Validate early and exit with a clear message:

```bash
os__require_root

if [[ ! "$VERSION" =~ ^[0-9]+\.[0-9]+\.[0-9]+$|^latest$ ]]; then
  logging__error "Invalid version: '${VERSION}'"
  exit 1
fi
```

Always call `os__require_root` near the top of any script that installs system packages or writes to system directories.

### Main logic

Write idempotent logic wherever possible — check whether work has already been done before doing it:

```bash
if command -v mytool > /dev/null 2>&1; then
  logging__info "mytool already installed — skipping."
else
  logging__install "Installing mytool..."
  # ... installation steps ...
fi
```

When writing new logic, ask: could this be useful in more than one feature, or does it encapsulate a non-trivial detail easy to get wrong? If yes, add it to `lib/` and run `just sync` to propagate it.

End the script with a success message:

```bash
logging__success "my-feature setup complete."
```

### OS package installation

Use `ospkg__run` for manifests and `ospkg__install` for individual packages:

- **`ospkg__run --manifest <file>`** — full pipeline: detect PM, update package lists, install packages from the manifest, run pre/post scripts, clean cache.
- **`ospkg__install <pkg>...`** — installs named packages directly (with idempotency check on APT/DNF).

When calling both in the same script, use `--keep_cache` on `ospkg__run` and call `ospkg__clean` once at the end:

```bash
ospkg__run --manifest "$_MANIFEST" --skip_installed --keep_cache
ospkg__install some-extra-package
ospkg__clean
logging__success "my-feature setup complete."
```

### Static files

Place static files under `features/<feature-id>/files/`. Reference them in `install.bash` via `_FILES_DIR`:

```bash
_FILES_DIR="${_BASE_DIR}/files"
cp "${_FILES_DIR}/my-config.conf" /etc/my-config.conf
```

During sync, `files/` is copied to `src/<feature-id>/files/` (not symlinked).

## Shared library reference

All library files live in `lib/` and are synced to `_lib/` in each feature. Source them from `$_SELF_DIR/_lib/<file>.sh`.

Each module uses a guard variable (`_LIB_<NAME>_LOADED`) to prevent double-sourcing. Every public function is covered by the bats unit suite under `test/unit/`. Run `just test-unit` to verify changes locally before pushing.

**Multi-value conventions:** many helpers return multiple logical items as one stdout line per item (empty list → no output). This composes naturally with pipes, `while read -r`, and `mapfile`.

> **Always check here before implementing something from scratch.** If a function does what you need, use it. If you are writing logic that could benefit other features, add it to `lib/` instead of keeping it inline.

#### ospkg manifest format

Manifests are written in YAML. See [`features/install-os-pkg/manifest.schema.json`](../../features/install-os-pkg/manifest.schema.json) and the [ospkg manifest instructions](../../.github/instructions/ospkg-manifests.instructions.md) for the complete reference. A brief overview:

```yaml
# Optional global condition — skips the entire manifest if false
when: {pm: apt}

# Signing keys fetched before repos/packages
keys:
  - url: https://example.com/key.gpg
    dest: /usr/share/keyrings/example.gpg

# Repository lines to add (PM-native format; ${deb_arch} etc. are substituted at runtime)
repos:
  - content: "deb [arch=${deb_arch} signed-by=...] https://repo.example.com stable main"

# Unconditional packages (all PMs)
packages:
  - git
  - name: curl
    when: {pm: apt}
  - name: htop
    version: "3.2.1"

# Per-PM package blocks
apt:
  packages:
    - build-essential
brew:
  casks:
    - visual-studio-code

# Shell commands run before and after package installation
prescripts: |
  install -d /opt/myapp
scripts: |
  ldconfig
```

Available `when` condition keys: `pm` (detected PM: `apt`, `brew`, `dnf`, `apk`, `yum`, `zypper`, `pacman`), `kernel` (`linux`/`darwin`), `arch`, `deb_arch`, `id`, `id_like`, `version_id`, `version_codename`, and any other `/etc/os-release` field.

```bash
  VERSION="$(github__latest_tag owner/repo)" || { logging__error "Failed to resolve version."; exit 1; }
fi

# Download a release asset with SHA-256 verification
github__fetch_release_asset_tarball owner/repo "$tag" "tool-linux-amd64.tar.gz" /tmp/tool.tar.gz
```
